# Blade Storm

协作式头脑风暴与设计探索工具，通过渐进式对话将模糊想法精炼为可落地的技术方案。

## 快速开始

```bash
# 带想法启动
/blade-storm 我想给系统加一个消息通知中心

# 直接开始对话
/blade-storm

# 讨论具体问题
/blade-storm 数据导出功能要不要做成异步的
```

## 工作流程

```
想法输入 → 理解上下文 → 逐步提问 → 方案探索 → 渐进式设计 → 文档输出
                                                                  ↓
                                                         衔接 blade-spec
```

### 阶段一：理解上下文

自动扫描项目状态，了解现有架构、相关模块和近期提交，为对话建立基础。

### 阶段二：逐步提问

沿「核心目标 → 用户角色 → 关键场景 → 边界约束 → 成功标准」路径逐一提问：

```
"这个功能主要是给谁用的？"
"使用这个功能的人是 (A) 管理员 (B) 普通用户 (C) 两者都有？"
"数据量大概是什么级别？(A) 几百条 (B) 几万条 (C) 百万级以上"
```

### 阶段三：方案探索

提出 2-3 种实现思路，推荐方案在前：

```
**方案一：事件驱动 + 消息队列** ⭐ 推荐
- 优点：解耦彻底，支持高并发
- 代价：引入 MQ 依赖，运维复杂度上升

**方案二：数据库轮询**
- 优点：实现简单，无额外依赖
- 代价：实时性差，数据库压力大
```

### 阶段四：渐进式设计

分段呈现设计（每段 200-300 字），逐段确认：

```
### 数据模型

[200-300 字的设计描述]

---
这部分设计看起来合理吗？有什么需要调整的地方？
```

### 阶段五：文档输出

汇总生成正式设计文档，写入 `docs/plans/YYYY-MM-DD-<topic>-design.md`。

## 对话节奏感知

| 用户信号 | 解读 | 行动 |
|---------|------|------|
| 回答详细，追问细节 | 希望深入探讨 | 继续提问，不急于收敛 |
| 回答简短，如"行"/"可以" | 希望加快节奏 | 减少提问，尽快进入方案 |
| "差不多了"/"你先说说" | 提问足够了 | 立即转入方案探索 |
| "等等"/"不对" | 前面有误解 | 回溯到出问题的地方 |
| "太复杂了"/"简单点" | 设计过度了 | 砍掉非必要部分 |
| "能不能快点" | 过程太长 | 合并剩余段落一次输出 |

## 核心原则

- **一次一问**，不用信息轰炸让人思考停滞
- **选择优于填空**，能给选项就给选项
- **YAGNI 原则**，只保留当下确定需要的
- **方案先行**，先对比 2-3 种思路再定细节
- **分段确认**，设计文档逐段验证
- **随时回溯**，用户说"不对"立刻停下来重新理清

## 与其他 Blade 技能的衔接

```
blade-storm（想法 → 设计）
    ↓
blade-spec（设计 → 任务拆解 → 自动开发）
    ↓
blade-design（代码生成）+ blade-commit（提交）
```

- 设计方案明确后 → `/blade-spec` 进入规范驱动开发
- 只需快速生成代码 → `/blade-design` 直接生成
- 不在 blade-storm 阶段写代码，产出是设计文档
