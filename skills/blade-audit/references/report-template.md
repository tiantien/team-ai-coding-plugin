## 4. 审计发现

按维度分组，每组内按严重级别降序排列。

### 4.1 安全审计

#### 🔴 [SEC-03] 删除接口缺少鉴权控制

**位置**：`blade-auth/src/main/java/.../controller/UserController.java:156`

**问题描述**：
`remove` 方法未配置 `@PreAuth` 注解，任何已登录用户均可调用删除接口，存在越权删除风险。

**影响分析**：
该接口允许通过 ID 批量删除用户数据，若被恶意调用可能导致关键数据丢失。在多租户环境下，还可能发生跨租户数据删除。

**相关代码**：
```java
// UserController.java:156
@PostMapping("/remove")
@Operation(summary = "删除用户")
public R remove(@RequestParam String ids) {
    return R.status(userService.removeByIds(Func.toLongList(ids)));
}
```

**修复建议**：
添加权限注解，并在 Service 层增加数据归属校验：
```java
@PostMapping("/remove")
@PreAuth(RoleConstant.HAS_ROLE_ADMIN)
@Operation(summary = "删除用户")
public R remove(@RequestParam String ids) {
    return R.status(userService.removeByIds(Func.toLongList(ids)));
}
```

## 6. 系统性问题

如果交叉分析发现了模式性问题（同类问题在多处重复），在此处集中说明。

### 系统性问题 1：全局缺少参数校验

**影响范围**：12 个 Controller 中的 28 个接口
**问题模式**：Controller 方法的 DTO 参数均未使用 `@Valid` 注解，DTO 类也缺少约束注解
**根因分析**：项目初期未建立参数校验规范，后续开发沿袭了无校验的写法
**统一解决方案**：
1. 为所有 DTO 添加 JSR-380 校验注解
2. 在 Controller 方法参数上添加 `@Valid`
3. 配置全局异常处理器统一处理 MethodArgumentNotValidException

## 单条发现的标准格式

每条审计发现必须包含以下六要素：

```markdown
#### {严重级别图标} [{检查项ID}] {问题摘要}

**位置**：`{文件相对路径}:{行号}`

**问题描述**：
{用 1-2 句话说明发现了什么问题}

**影响分析**：
{说明这个问题如果不修复，可能造成什么后果}

**相关代码**：
```{language}
// {文件名}:{行号}
{相关代码片段，通常 3-10 行，包含足够上下文}
```

**修复建议**：
{给出具体的修复方案，最好附带修复后的代码示例}
```

### 要求

- **位置**必须精确到文件和行号
- **影响分析**要从业务和系统角度解释，不能只说"不符合规范"
- **修复建议**要具体可操作，而非泛泛的"建议优化"
- **相关代码**应包含足够上下文帮助理解，但不要粘贴整个文件
