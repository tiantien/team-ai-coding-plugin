# blade-audit — BladeX 专业代码审计工具

从资深架构师视角对 BladeX 工程进行全方位代码审计，涵盖安全、架构、规范、性能、逻辑、质量六大维度，自动生成专业审计报告。

## 功能概述

blade-audit 是 BladeX 工具链中的质量守门人。它不是简单的代码风格检查，而是站在架构师高度，审视代码的设计合理性、安全健壮性和工程规范性。每一条发现都附带精确定位、影响分析和修复建议，帮助团队在上线前发现并消除风险。

### 核心能力

- **六维度全覆盖**：安全漏洞、逻辑健壮性、架构合规、框架规范、性能隐患、代码质量
- **三种审计模式**：Commit 审计、设计文档对比审计、目录全量审计
- **三级审计深度**：快速扫描（shallow）、标准审计（standard）、深度审计（deep）
- **精确定位**：每条发现精确到文件路径和行号
- **BladeX 原生**：内置 BladeX 框架规范检查，了解 Boot/Cloud/Links 三种工程类型
- **专业报告**：结构化审计报告，含评分、风险清单、改进路线图

## 快速开始

### Commit 审计 — 审查最近的提交

```
/blade-audit --commit HEAD~3..HEAD
```

审计最近 3 个 commit 的代码变更，检查是否引入了安全风险或质量问题。

### 目录审计 — 全量检查某个模块

```
/blade-audit --module blade-auth
```

对 blade-auth 模块进行标准深度的全量审计。

### 设计审计 — 对比设计与实现

```
/blade-audit --design ./docs/user-module-spec.md ./blade-service/blade-system
```

对比设计文档与实际实现，检查功能覆盖度和接口一致性。

### 聚焦审计 — 只关注安全维度

```
/blade-audit --module blade-auth --focus security
```

仅对 blade-auth 模块进行安全维度的深入审计。

### 深度审计 — 发布前全面检查

```
/blade-audit --depth deep
```

对整个工程进行深度审计，包含数据流分析和交叉关联分析。

## 三种审计模式

### 模式一：Commit 审计

**用途**：审查特定 Git 提交中的代码变更

**命令格式**：
```
/blade-audit --commit [commit-range]
```

**支持的 commit-range 格式**：
| 格式 | 含义 | 示例 |
|------|------|------|
| `<hash>` | 单个 commit | `abc1234` |
| `<hash1>..<hash2>` | 范围 | `abc1234..def5678` |
| `HEAD~N..HEAD` | 最近 N 个 | `HEAD~3..HEAD` |
| 省略 | 最近一次提交 | — |

**审计内容**：
- 对新增/修改的代码执行六维度审计
- 评估变更的影响范围和回归风险
- 检查 commit message 质量和提交规范

**适用场景**：
- 日常 Code Review
- 合并到主分支前的质量把关
- 提交后的回顾检查

### 模式二：设计审计

**用途**：对比设计文档与实际代码实现的一致性

**命令格式**：
```
/blade-audit --design <design-doc-path> [code-path]
```

**审计内容**：
- 功能覆盖度：设计的每个功能点是否都已实现
- 接口一致性：API 的 URL、参数、返回值是否与设计一致
- 数据模型一致性：表结构和实体是否与设计匹配
- 业务规则一致性：代码逻辑是否符合设计描述
- 偏差分析：区分"合理演进"和"遗漏/错误"

**适用场景**：
- 使用 `/blade-spec` 规划后的验收环节
- 需求评审后检查实现完整性
- 外包交付物的验收审计

### 模式三：目录审计

**用途**：对工程或模块进行全量代码审计

**命令格式**：
```
/blade-audit [path] [options]
```

**可选参数**：
| 参数 | 说明 | 示例 |
|------|------|------|
| `--module <name>` | 仅审计指定模块 | `--module blade-auth` |
| `--focus <dim>` | 聚焦特定维度 | `--focus security` |
| `--depth <level>` | 审计深度 | `--depth deep` |

**`--focus` 可选值**：
| 值 | 维度 |
|----|------|
| `security` | 安全审计 |
| `robustness` | 逻辑健壮性 |
| `architecture` | 架构合规 |
| `convention` | 框架规范 |
| `performance` | 性能隐患 |
| `quality` | 代码质量 |

**适用场景**：
- 新成员接手项目时的代码摸底
- 版本发布前的全面质量检查
- 技术债务评估和整改规划

## 六大审计维度

### 1. 安全审计 (security) — 权重 25%

从 OWASP Top 10 和 BladeX 安全框架的角度检查：

| 检查项 | 说明 |
|--------|------|
| SQL 注入 | MyBatis `${}` 拼接、手动 SQL 拼接 |
| XSS | 用户输入未经转义直接输出 |
| 接口鉴权 | 缺少 `@PreAuth`、匿名可访问的写接口 |
| 敏感数据 | 明文密码、硬编码密钥、日志泄露 |
| 越权访问 | 水平越权、垂直越权、缺少数据归属校验 |
| 文件上传 | 类型未校验、路径穿越、大小未限制 |
| 租户隔离 | 多租户数据泄露风险 |
| 序列化安全 | 不安全的反序列化配置 |

### 2. 逻辑健壮性 (robustness) — 权重 20%

从系统稳定性和正确性角度检查：

| 检查项 | 说明 |
|--------|------|
| 空指针防护 | 链式调用 NPE、查询结果未判空 |
| 事务管理 | 多表写操作无事务、事务自调用失效 |
| 并发安全 | 竞态条件、先查后改无保护 |
| 异常处理 | 吞异常、捕获过宽、异常信息丢失 |
| 数据一致性 | 跨表操作一致性、级联删除 |
| 幂等性 | 关键写接口的重复提交防护 |
| 边界条件 | 分页上限、空集合、数值溢出 |

### 3. 架构合规 (architecture) — 权重 20%

从系统架构和设计原则角度检查：

| 检查项 | 说明 |
|--------|------|
| 分层纪律 | Controller 直调 Mapper、Service 操作 Request |
| 模块耦合 | Cloud 架构下跨模块直接依赖 |
| 依赖方向 | 下层反向依赖上层 |
| 职责单一 | God Class、超长方法 |
| API 设计 | RESTful 规范、接口粒度 |
| 包结构 | BladeX 标准分包规范 |

### 4. 框架规范 (convention) — 权重 15%

从 BladeX 框架特定规范角度检查：

| 检查项 | 说明 |
|--------|------|
| 实体层次 | Entity/DTO/VO 使用是否规范 |
| 服务接口 | IXxxService 接口定义 |
| 统一返回 | `R<T>` 返回格式 |
| MyBatis-Plus | LambdaQueryWrapper、批量操作 |
| 租户处理 | tenant_id 规范、租户插件使用 |
| 数据库字段 | 必要基础字段、命名规范 |
| Wrapper 转换 | BaseEntityWrapper 对象转换 |
| API 文档 | Swagger/Knife4j 注解完整性 |

### 5. 性能隐患 (performance) — 权重 10%

从系统性能和资源使用角度检查：

| 检查项 | 说明 |
|--------|------|
| N+1 查询 | 循环内数据库查询 |
| 全表扫描 | 查询条件未命中索引 |
| 内存风险 | 全表加载、大对象循环创建 |
| 资源泄漏 | 流/连接未正确关闭 |
| 批量操作 | 逐条操作代替批量 |
| 缓存策略 | 热点数据无缓存 |
| 索引策略 | 缺少必要索引 |

### 6. 代码质量 (quality) — 权重 10%

从代码可读性和工程实践角度检查：

| 检查项 | 说明 |
|--------|------|
| 命名规范 | 类/方法/变量命名语义清晰度 |
| 方法复杂度 | 行数、圈复杂度、嵌套层数 |
| 代码重复 | 复制粘贴代码、可提取的公共逻辑 |
| 魔法值 | 未解释的硬编码数字/字符串 |
| 注释质量 | 过时注释、注释掉的代码 |
| 日志规范 | 日志级别、占位符使用 |
| 过时 API | 使用已弃用的方法或过时编码方式 |

## 严重级别

| 级别 | 图标 | 含义 | 处理要求 |
|------|------|------|----------|
| 严重 | 🔴 | 安全漏洞、数据损坏风险 | 上线前必须修复 |
| 重要 | 🟠 | 稳定性、可维护性风险 | 当前迭代应修复 |
| 建议 | 🟡 | 质量和可读性改进 | 建议改进 |
| 提示 | 🟢 | 最佳实践建议 | 参考信息 |

## 审计深度

| 深度 | 覆盖级别 | 分析范围 | 适用场景 |
|------|----------|----------|----------|
| shallow | 仅 🔴 | 高优先级项快速扫描 | 日常自检 |
| standard | 🔴🟠🟡 | 全维度核心检查项 | Code Review、迭代交付 |
| deep | 🔴🟠🟡🟢 | 含数据流分析、交叉关联 | 版本发布、技术债务评估 |

## 评分体系

### 维度评分

```
维度合规分 = (通过检查项数 / 适用检查项总数) × 100
```

### 综合评分（加权平均）

```
安全审计：25% + 逻辑健壮性：20% + 架构合规：20%
+ 框架规范：15% + 性能隐患：10% + 代码质量：10%
```

### 评分等级

| 等级 | 分数 | 含义 |
|------|------|------|
| A | 90-100 | 优秀，可放心上线 |
| B | 75-89 | 良好，建议修复重要问题后上线 |
| C | 60-74 | 合格，需修复所有严重和重要问题 |
| D | 40-59 | 不合格，需较大范围整改 |
| F | <40 | 严重不合格，建议重新设计 |

## 审计报告结构

生成的审计报告包含以下章节：

1. **审计概览** — 一段话总结，关键数字和核心发现
2. **审计信息** — 时间、范围、模式、深度、工程类型
3. **评分总览** — 六维度评分表 + 综合评分 + 文本雷达图
4. **审计发现** — 按维度分组，含位置、描述、影响、代码、修复建议
5. **关键风险清单** — 所有 🔴🟠 问题的集中清单
6. **系统性问题** — 跨维度的模式性问题分析
7. **改进路线图** — 立即修复 → 短期改进 → 长期优化

## 执行流程

```
Phase 1: 环境准备与范围确定
  ├── 解析输入参数
  ├── 识别工程类型 (Boot/Cloud/Links)
  ├── 确定审计范围
  └── 向用户确认范围 ← 用户确认后继续
          │
Phase 2: 代码采集与预分析
  ├── 按模式采集代码
  ├── 构建模块依赖图
  └── 统计基础指标
          │
Phase 3: 逐维度深度审计
  ├── 安全审计 (SEC)
  ├── 逻辑健壮性 (ROB)
  ├── 架构合规 (ARC)
  ├── 框架规范 (CON)
  ├── 性能隐患 (PRF)
  └── 代码质量 (QLT)
          │
Phase 4: 交叉分析与风险评估
  ├── 关联分析（同文件跨维度）
  ├── 模式识别（系统性问题）
  ├── 风险聚合（关键路径）
  └── 合规评分计算
          │
Phase 5: 报告生成
  ├── 填充报告模板
  ├── 生成改进路线图
  └── 输出并展示报告
```

## 安全保证

- **只读操作**：审计过程绝不修改任何源代码或 Git 历史
- **精确定位**：每条发现附带文件路径和行号
- **不做臆断**：不确定的问题标注为「待确认」
- **分级客观**：严格按定义分级，不人为调整

## 与其他 Skill 协作

| 工作流 | 组合方式 |
|--------|----------|
| 规范驱动开发 | `/blade-spec` → 开发 → `/blade-audit --design` 验收 |
| 审计后同步修复 | `/blade-audit` → 修复 → `/blade-sync` 同步到镜像工程 |
| 差异深入审计 | `/blade-compare` 发现差异 → `/blade-audit` 审计差异代码 |
| 提交后审查 | `/blade-commit` → `/blade-audit --commit HEAD` |
| 方案验证 | `/blade-storm` 讨论 → 落地 → `/blade-audit` 验证 |
