# Blade Spec

规范驱动开发（Spec-Driven Development）工作流，将复杂需求自动拆解为结构化规范文档，再逐任务驱动自动开发。

## 核心理念

**先谋定，后行动。** 面对复杂功能开发，不再"边聊边写"，而是通过三阶段规范文档锚定上下文，让 AI 全程有据可循。

```
一句话需求 → 需求分析 → 技术设计 → 任务拆解 → 逐任务自动执行 → 完成总结
```

## 快速开始

```bash
# 启动新 Spec — 交互模式，每阶段需审批
/blade-spec 多模型对话管理功能，支持多轮会话、模型切换和历史记录

# 启动新 Spec — 自动模式，全流程一次性完成，无需介入审批
/blade-spec --auto 多模型对话管理功能，支持多轮会话、模型切换和历史记录

# 继续未完成的 Spec
/blade-spec continue chat-management

# 执行下一个任务
/blade-spec next

# 查看所有 Spec 状态
/blade-spec status
```

## 工作流程

**交互模式（默认）：** 每个阶段生成文档后暂停等待用户审阅，审批通过后进入下一阶段。

```
  ┌─────────────────────────────────────────────────────────────┐
  │                                                             │
  │  📋 需求分析          📐 技术设计          📝 任务拆解      │
  │  requirements.md  →  design.md         →  tasks.json       │
  │       ↓ 审批              ↓ 审批              ↓ 审批       │
  │                                                             │
  │  🔨 逐任务执行（自动更新 tasks.json 状态）                  │
  │  T-001 ✅ → T-002 ✅ → T-003 🔄 → T-004 ⏳ → ...          │
  │       ↓                                                     │
  │  🎉 完成总结 → result.md                                    │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘
```

**自动模式（--auto）：** 跳过所有审批门控，从需求分析到代码生成一口气完成。

```
  ┌─────────────────────────────────────────────────────────────┐
  │                                                             │
  │  📋 需求分析  →  📐 技术设计  →  📝 任务拆解  →  🔨 连续执行 │
  │  （每阶段输出一行进度，不暂停）                              │
  │       ↓                                                     │
  │  🎉 完成总结 → result.md                                    │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘
```

自动模式适合需求已经想清楚、希望快速出完整骨架代码的场景。构建失败和无法完成的任务仍会中断报告。

### 三阶段产物

| 阶段 | 产物 | 内容 |
|---|---|---|
| 需求分析 | `requirements.md` | 需求原文（润色版）、用户故事、GIVEN/WHEN/THEN 验收标准、范围界定 |
| 技术设计 | `design.md` | 架构设计、数据模型、API 定义、Mermaid 流程图 |
| 任务拆解 | `tasks.json` | 有序任务列表，每个任务关联需求编号，带依赖和状态追踪 |

### 任务状态追踪

`tasks.json` 实时记录每个任务的执行状态：

```json
{
  "id": "T-005",
  "title": "创建 ILlmConversationService 接口和实现",
  "status": "completed",
  "requirement_ids": ["R-001", "R-003"],
  "files_created": [
    ".../llm/business/service/ILlmConversationService.java",
    ".../llm/business/service/impl/LlmConversationServiceImpl.java"
  ],
  "result": "实现对话管理服务，继承 BaseServiceImpl，支持创建/查询/删除会话和多模型切换"
}
```

支持中断恢复 — 随时可通过 `/blade-spec continue` 从上次断点继续。

## 适用场景

| 场景 | 适合 | 不适合 |
|---|---|---|
| 从零开发完整功能模块 | ✅ | |
| 复杂系统设计与架构规划 | ✅ | |
| 大型代码重构 | ✅ | |
| 修复牵连多模块的 Bug | ✅ | |
| 简单的单文件修改 | | ❌ 直接对话即可 |
| 快速查询或问答 | | ❌ 直接对话即可 |

## 特性

- **BladeX 体系适配**：内置 Boot 单体与 Cloud 微服务两种工程模式的最佳实践，同时可适配其他技术栈
- **上下文锚定**：规范文档作为"单一事实来源"，防止 AI 在长对话中偏离需求
- **可追溯性**：每个任务 → 需求编号 → 验收标准，全链路可追踪
- **断点续传**：基于 `spec.json` 状态管理，支持任意中断后恢复
- **自动构建验证**：每完成一组逻辑相关任务后自动执行构建检查
- **连续执行模式**：说"全部执行"即可自动跑完所有任务
- **自动模式**：`--auto` 参数跳过所有审批门控，从需求到代码一次性全自动完成

## 文件结构

```
blade-spec/                        # Skill 定义
├── README.md                      # 本文件
├── SKILL.md                       # Skill 主指令
└── references/
    ├── templates.md               # 各阶段文档模板（spec.json / requirements / design / tasks / result）
    └── phase-guide.md             # 阶段转换规则、质量检查清单、异常处理

{project}/.claude/blade-spec/      # 运行时产物（按 Spec 分文件夹）
└── chat-management/
    ├── spec.json                  # 元数据与状态
    ├── requirements.md            # 需求文档
    ├── design.md                  # 技术设计
    ├── tasks.json                 # 任务清单与进度
    └── result.md                  # 完成总结
```
